# Security Audit: latte-java HTTP Server — 2026-04-20

Consolidated high-confidence findings after parallel verification. Only findings with ≥8 confidence after filtering are included. DoS-only, lack-of-hardening, and handler-footgun issues were excluded from scope.

---

## Vuln 1: HTTP Request Smuggling via non-strict Transfer-Encoding parsing

- **File:line**: `server/internal/HTTPWorker.java:432-458`, `server/HTTPRequest.java:618-629`, `server/io/HTTPInputStream.java:159-207`
- **Severity**: HIGH
- **Category**: `request_smuggling`
- **Confidence**: 10/10

**Description**: `HTTPWorker.validatePreamble` unconditionally strips `Content-Length` whenever any `Transfer-Encoding` header is present (`if (getHeader(TransferEncoding) == null) { ... } else { setContentLength(null); removeHeader(ContentLength); }`). Downstream, `HTTPInputStream.initialize` only routes the body through `ChunkedInputStream` when `isChunked()` returns true, and `isChunked()` requires `"chunked".equalsIgnoreCase(getTransferEncoding())` — an exact, trimmed match. Because `RequestPreambleState.HeaderValue` accepts trailing SP/TAB/obs-text into the stored value (`HTTPTools.isValueCharacter` accepts `\t`, ` `, `\n`, and 0x80–0xFF), many legal-looking variants bypass the exact match: `chunked ` (trailing space), `chunked\t`, `chunked, identity`, `identity`, `xchunked`. For all of these, CL is discarded and the body reads `InputStream.nullInputStream()` — the declared body bytes are left unread in the socket and become the preamble of the next keep-alive request.

**Exploit scenario**: Attacker behind a front-end proxy that honors `Content-Length` (or parses `chunked, identity` per RFC 9112 §6.1 as chunked) sends:
```
POST / HTTP/1.1
Host: victim
Transfer-Encoding: chunked, identity
Content-Length: 44

0

GET /admin HTTP/1.1
Host: victim
X: X
```
The front-end sees chunked framing and forwards both. Latte strips `Content-Length`, `isChunked()` returns false, 0 body bytes consumed. On keep-alive, the trailing bytes parse as `GET /admin ...` attributed to the next client's connection — classic TE desync enabling auth bypass, cache poisoning, or request hijacking.

**Fix**: In `validatePreamble`, if `TransferEncoding` is present but `isChunked()` is false, return 400 and close. Reject requests carrying both `Content-Length` and `Transfer-Encoding` rather than silently stripping CL. Trim/token-parse the TE value strictly per RFC 9112. Reject multiple `Transfer-Encoding` headers.

## Vuln 2: Chunk size integer truncation enables request smuggling

- **File:line**: `io/ChunkedInputStream.java:105` (cast), `:281-297` (terminal-chunk state)
- **Severity**: HIGH
- **Category**: `request_smuggling` / `integer_overflow`
- **Confidence**: 10/10

**Description**: `chunkSize = (int) Long.parseLong(headerSizeHex, 0, headerSizeHex.length(), 16);`. `chunkSize` is declared `int`. A chunk size ≥ 2³² truncates: `0x100000000L` casts to `0`, and the state machine then enters `Chunk` with `length == 0`, immediately treating it as the terminating chunk (`Chunk.next` returns `Trailer`/`Complete` when `length == 0`). There is no upper-bound check on `headerSizeHex` length.

**Exploit scenario**: Attacker sends a chunked body:
```
Transfer-Encoding: chunked

100000000
<arbitrary-smuggled-HTTP-request-bytes>
0

```
A compliant upstream proxy parses the 4 GiB chunk and forwards all bytes. Latte truncates `chunkSize` to `0`, declares end-of-body, and leaves the smuggled bytes in the socket to be parsed as the next pipelined request.

**Fix**: Parse as `long`, validate `0 ≤ size ≤ configuredMax` before narrowing to `int`, throw `ChunkException` on overflow. Bound `headerSizeHex.length()` (≤ 16 hex chars).

## Vuln 3: Bare LF accepted in header values (smuggling auxiliary)

- **File:line**: `util/HTTPTools.java:143-147` (`isValueCharacter`), `util/RequestPreambleState.java:188-204` (HeaderValue state)
- **Severity**: HIGH
- **Category**: `header_injection` / `request_smuggling`
- **Confidence**: 10/10

**Description**: `isValueCharacter` explicitly allows `'\n'` (0x0A) and obs-text (0x80–0xFF). The `HeaderValue` state transitions out only on `'\r'`; a bare `\n` is appended to the value buffer and parsing continues as if still inside the same header value. RFC 9112 §2.2 mandates rejecting or replacing bare CR/LF in field values; this parser does neither.

**Exploit scenario**: Any header containing a bare LF desynchronizes from front-end parsers that split on LF. Combined with Vuln 1, `Transfer-Encoding: chunked\nX: Y` stores the value `chunked\nX: Y`; `equalsIgnoreCase("chunked")` returns false → Vuln 1 desync triggered. Also desyncs duplicate-header-detection in `validatePreamble` because front-end sees two headers while Latte sees one.

**Fix**: Remove `'\n'` from `isValueCharacter`. Require CRLF to terminate `HeaderValue`; reject bare `\r` or `\n` anywhere inside a header value. Optionally reject obs-text (0x80–0xFF) in safety-critical headers.

## Vuln 4: HTTP Response Splitting via unescaped response APIs

- **File:line**:
  - `server/HTTPResponse.java:63-69` (`addHeader`), `:310-316` (`setHeader`)
  - `server/HTTPResponse.java:282-289` (`sendRedirect`)
  - `server/HTTPResponse.java:196-198` (`setStatusMessage`)
  - `Cookie.java:402-431` (`toResponseHeader`), constructors at `:68, :73`, `addAttribute` at `:244`
  - `util/HTTPTools.java:397-422` (`writeResponsePreamble`), `:487-496` (`writeStatusLine`)
- **Severity**: HIGH
- **Category**: `response_splitting` / `header_injection`
- **Confidence**: 10/10

**Description**: No point in the response-generation chain validates `\r`, `\n`, or `\0` in header names, header values, cookie fields (name/value/domain/path/attributes), status messages, or redirect URIs. `writeResponsePreamble` writes `name + ": " + value + CRLF` byte-for-byte. `Cookie.toResponseHeader()` concatenates all fields with `StringBuilder.append` with no escaping. `sendRedirect` forwards the URI straight to `setHeader`. Peer libraries (Jetty, Tomcat, Netty) all reject CRLF at the container boundary; this library does not.

**Exploit scenario**: Any application that passes untrusted input into these APIs is vulnerable. Example: `response.sendRedirect(request.getParameter("next"))` with `next=/home%0d%0aSet-Cookie:%20sid=evil%0d%0aContent-Length:%200%0d%0a%0d%0aHTTP/1.1%20200%20OK...` emits a 302 followed by a fully-smuggled response that downstream caches and browsers will treat as legitimate — session fixation, cache poisoning, reflected XSS via injected body. Same class of attack reachable via `setHeader`, `addHeader`, cookie fields, and `setStatusMessage`.

**Fix**: At a single choke point (recommended: `setHeader`/`addHeader`/`sendRedirect`/`setStatusMessage`/`Cookie` setters and `toResponseHeader`), throw `IllegalArgumentException` if the value contains `\r`, `\n`, or `\0`. Validate header names against RFC 7230 tchar. Validate cookie name per RFC 6265 token; reject `;` in value/path/domain unless properly quoted.

## Vuln 5: MultipartStream.reload() pointer corruption

- **File:line**: `io/MultipartStream.java:382`
- **Severity**: MEDIUM
- **Category**: `parser_desync` / `buffer_corruption`
- **Confidence**: 9/10

**Description**: The `reload` loop contains `start += end;` where the intent is plainly `start += read;` (the newly-read byte count, not the cumulative). On a single-iteration read (`start=0, end=0`), `start += end` coincidentally matches `start += read` — masking the bug in tests. On iteration 2+, `start` grows roughly as `2·cumulative_read` instead of the true write offset, overshooting `buffer.length`. The next `input.read(buffer, start, buffer.length - start)` gets a negative third argument and throws `IndexOutOfBoundsException`, or (when `start == buffer.length`) invokes zero-length-read behavior. Multi-iteration reload is attacker-reachable: TLS record boundaries, TCP segmentation, or a slow/trickling client forces it routinely; the multipart boundary (which drives `minimumToLoad = boundaryLength + 2`) is attacker-controlled via the request's `Content-Type`.

**Exploit scenario**: Uploading a multipart body with segmented TCP delivery raises `IndexOutOfBoundsException` mid-parse, terminating the request with a 500 and potentially leaving state inconsistent. The stronger content-confusion chain (a boundary match synthesized inside a victim's part data) depends on the specific read pattern and is less reliable but plausible.

**Fix**: Change `start += end` to `start += read` on `MultipartStream.java:382`. Add `assert start <= buffer.length`.

## Verified NOT vulnerable (refuted during filtering)

- **Duplicate `Content-Length` desync**: `validatePreamble` uses the raw header-list size (`requestedContentLengthHeaders.size() != 1` → 400) and parse-null contentLength with a present CL header is also caught by `contentLength == null || contentLength < 0`. Not exploitable.
- **Multipart filename traversal**: Library never uses the client filename for on-disk paths (`Files.createTempFile` only). Matches standard Servlet/Commons contract; lack-of-hardening, not a vuln.
- **TLS `SSLContext.getInstance("TLS")` without `setEnabledProtocols`**: JDK 21 (project-required) disables TLSv1.0/1.1 via `jdk.tls.disabledAlgorithms` by default and has no weak cipher suites enabled. Not concretely exploitable.
- **`SecurityTools` `getBytes()` platform default charset**: PEM content is strictly ASCII; identically encoded in every ASCII-superset charset. Java 21 default is UTF-8 anyway.
- **Chunked terminator laxness (`0\n`)**: REFUTED — the state machine strictly requires `ChunkSize → CR → LF`; bare LF throws `ParseException`.
- **Response TE double-chunking via handler-set `Transfer-Encoding: gzip`**: Reachable only by handler code inside the app trust boundary; not an attacker-reachable primitive.

## Prioritized fix order

1. **Vuln 4** **FIXED** (response splitting) — single choke point addition; broadest blast radius across every app using the library.
2. **Vuln 1** (TE smuggling) — protocol-level compliance; fix `validatePreamble` to require strict `chunked` and reject CL+TE coexistence.
3. **Vuln 3** (bare LF in header values) — underlies Vuln 1 bypass; one-line fix in `isValueCharacter` plus state transition tightening.
4. **Vuln 2** (chunk size overflow) — bound/validate before int-narrowing.
5. **Vuln 5** (multipart reload) — one-character fix: `end` → `read`.
